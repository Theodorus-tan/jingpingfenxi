三、编码开发规范（字节内部强制标准）
3.1 Go编码规范
1. 命名规范
  - 包名：全小写单数，无下划线，如user、task
  - 导出标识符：大驼峰（PascalCase），如UserService
  - 私有标识符：小驼峰（camelCase），如userID
  - 常量：大驼峰，如MaxRetryTimes
  - 错误变量：以Err开头，如ErrUserNotFound
  - 接口名：以er结尾，如Reader、Storer
2. 代码风格
  - 使用gofmt自动格式化代码
  - 函数长度不超过50行，单个文件不超过500行
  - 错误必须处理，禁止忽略错误（_）
  - 日志必须结构化，区分DEBUG/INFO/WARN/ERROR等级，生产环境禁止输出敏感信息
  - 测试文件与源文件同目录，以_test.go为后缀，使用Go原生测试框架
3. 接口规范
  - 严格遵循RESTful设计：GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）
  - 统一返回格式：{"code":200,"msg":"success","data":{}}
  - 状态码规范：200成功、400参数错误、401未授权、404资源不存在、500服务器错误
3.2 前端编码规范
1. 命名规范
  - 文件命名：短横线分隔法（kebab-case），如task-list.tsx
  - 组件命名：大驼峰法（PascalCase），与文件名一致，如TaskList.tsx
  - 变量/函数：小驼峰法（camelCase）
  - 常量：全大写+下划线，如MAX_PAGE_SIZE
2. 代码风格
  - 统一使用2个空格缩进
  - 字符串使用单引号，结尾不加分号
  - 组件必须写TypeScript类型定义，禁止使用any
  - 样式使用Arco Token系统，避免硬编码颜色、尺寸
  - 禁止在生产环境留下console.log调试代码
3. 组件规范
  - 优先使用Arco原生组件，禁止重复造轮子
  - 组件职责单一，一个组件只做一件事
  - 公共组件必须写注释和使用示例
  - 类型定义单独维护在interface.ts文件中
四、协作与工程化规范
4.1 Git工作流
字节内部采用Trunk-Based Development（主干开发）模式，竞赛可简化为：
- 主分支：main（保护分支，禁止直接提交）
- 功能分支：feat/xxx（从main切出，开发完成后合并回main）
- 修复分支：fix/xxx（紧急bug修复）
- 分支生命周期不超过2天，合并后立即删除
4.2 Git提交规范
遵循Conventional Commits标准，格式：类型: 描述
feat: 新增竞品分析任务创建接口
fix: 修复报告导出失败问题
docs: 更新部署手册
style: 调整页面样式
refactor: 重构用户服务逻辑
perf: 优化接口响应速度
test: 添加单元测试
chore: 更新依赖

4.3 代码评审
- 所有代码必须经过至少一人评审才能合并到主分支
- 评审重点：代码逻辑、命名规范、性能、安全性
- 合并时使用Squash and merge，保持提交历史整洁
4.4 CI/CD
- 提交代码后自动运行ESLint、单元测试
- 合并到主分支后自动构建Docker镜像
- 支持一键部署到测试环境
五、竞赛加分技巧
1. 优先使用字节自研技术：Hertz、Kitex、Eino、Arco Design，这是最大的加分项
2. 工程化完善：严格遵循目录规范、编码规范，有完整的CI/CD配置
3. 性能优化：展示Hertz的高性能优势，提供压测数据
4. 容器化部署：提供完善的Dockerfile和docker-compose.yml，一键启动
5. 文档齐全：按照字节内部标准交付8份正式PDF文档，文风遵循"结论先行、数据驱动、结构化表达"
