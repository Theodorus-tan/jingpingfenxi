# 竞品分析系统 — 业务架构与方法论白皮书

> **核心设计理念**：系统以“企业”为顶层载体，实现数据、任务与报告的全面隔离。基于业务场景沉淀“人肉分析师”方法论，并将其映射为 AI Agent 可执行的系统提示词（System Prompts），实现自动化的立体竞品研判。

---

## 一、 系统业务层级模型

本系统采用严格的三级业务架构，确保不同层级的业务数据互不干扰，分析任务有条不紊。

### 1.1 层级划分

1. **企业 (Enterprise)**
   - **定位**：系统的顶层载体，代表一个独立的商业实体或公司账号。
   - **作用**：隔离不同企业客户的数据。一个企业拥有全局的业务概览权限和资源配额。

2. **选品项目 (Selection Project)**
   - **定位**：企业下的具体业务线或产品研发计划。
   - **作用**：将分析目标收拢。例如“2025年智能水杯出海计划”。一个企业下可创建多个选品项目。

3. **独立分析工程 (Independent Analysis Engineering)**
   - **定位**：针对单一具体竞品的深度剖析任务。
   - **作用**：遵循**“一个选品对应一个独立分析工程”**的原则。工程内的数据源抓取、分析过程和生成的报告相互独立，确保深度与专注。

---

## 二、 三大核心分析脉络（3D分析模型）

在每个“独立分析工程”内部，固定搭载三大核心分析子工程，从不同维度对竞品进行立体化解剖。每个子工程对应特定的数据源和分析方法论。

### 2.1 维度一：Review 精准分析 (产品内核评估)
- **分析目标**：聚焦产品本身的真实市场反馈，提炼竞品短板、用户痛点，沉淀可借鉴的产品亮点。
- **核心数据源**：竞品独立站、亚马逊 (Amazon)、阿里国际站、京东、淘宝等电商平台的真实用户评论（Review）与问答（Q&A）。
- **分析方法论**：
  - **差评情感提取法**：统计一星/二星评论中的高频负面词汇，归纳致命缺陷。
  - **使用场景还原法**：从好评中提取用户实际使用场景，寻找“意料之外”的卖点。
  - **价格与价值锚定**：分析用户对价格的敏感度及性价比评价。

### 2.2 维度二：全网通用泛分析 (企业背景与战略研判)
- **分析目标**：宏观背景调研，建立对竞品的整体认知，研判其商业模式、成长路径和战略意图。
- **核心数据源**：百度/Google 搜索结果、企业官网、百科词条、新闻报道、行业白皮书等广谱公开信息。
- **分析方法论**：
  - **时间线回溯法**：梳理竞品的发展历程、重大融资节点和产品迭代时间线。
  - **核心团队背景穿透**：分析创始团队背景、技术基因和渠道资源。
  - **商业模式拆解画布**：梳理其盈利模式、核心资源、关键业务和客户关系。

### 2.3 维度三：股市金融生命力分析 (长期经营与抗风险能力评估)
- **分析目标**：量化评估竞品企业的经营健康度、资本认可度及发展潜力，判断其长期生命力和潜在风险。
- **核心数据源**：股票市场数据（如东方财富、雪球 API）、公开财报（年报/季报）、招股书。
- **分析方法论**：
  - **三表联动分析法**：交叉比对资产负债表、利润表、现金流量表，寻找利润背后的现金流支撑。
  - **研发投入占比趋势**：通过研发费用率评估其技术护城河的稳固程度。
  - **库存周转预警机制**：分析存货周转天数，判断是否存在压货风险或销售疲软。

## 三、 场景驱动的动态分析机制 (Scenario-Driven Analysis)

在上述 3D 分析模型的基础上，分析引擎必须具备“场景感知”能力。用户的商业诉求不同，Agent 的分析侧重点和输出结论必须动态调整。我们将主要场景划分为两大类：

### 3.1 场景 A：已有产品求改进 (Product Improvement)
- **用户画像**：企业已有同类产品在售，希望通过分析竞品寻找自身产品的迭代方向或调整营销策略。
- **分析侧重点**：
  - **Review**：重点提取竞品的“致命缺陷”作为我方产品的“护城河”优化方向；提取“高赞特性”进行特性对标 (Feature Comparison)。
  - **泛分析**：侧重于竞品的最新营销动作、渠道策略变化。
  - **股市金融**：关注竞品的短期盈利能力和研发投入变化，判断其下一代产品的发力点。

### 3.2 场景 B：无产品求入局 (Market Entry & Feasibility)
- **用户画像**：企业尚未涉足该领域，正在进行市场可行性调研，考虑是否立项研发该产品。
- **分析侧重点**：
  - **Review**：侧重宏观市场痛点，判断是否存在未被满足的蓝海需求；评估用户对价格的敏感度，用于定价策略。
  - **泛分析**：侧重行业壁垒分析、头部玩家护城河、专利技术封锁情况、整体市场容量。
  - **股市金融**：重点看该品类头部企业的整体毛利率水平、资金周转周期，评估入局的资金门槛和投资回报率 (ROI)。

## 四、 业务方法论与场景到 Agent 提示词（Prompt）的映射

将人类分析师的经验方法论结合**用户诉求场景**，转化为大模型易于理解和执行的系统提示词（System Prompt）。以下为各分析引擎的 Prompt 设计草案。

### 4.1 Review 分析师 Prompt 映射
```text
【角色设定】
你是一位资深的电商产品经理和消费者心理学专家，精通通过海量用户评论（Review）洞察产品优劣势与市场切入点。

【用户当前诉求场景】
{{ 场景变量：已有产品求改进 / 无产品求入局 }}

【输入数据】
电商平台抓取的竞品评论列表（包含星级、评论内容、时间）。

【执行任务（方法论映射）】
1. 差评归因（对应差评情感提取法）：提取所有3星及以下评价，分类汇总用户痛点。
2. 爽点提炼（对应使用场景还原法）：分析4-5星评价，提取用户提到最多的“超出预期”的功能点或使用场景。
3. 动态战略输出（基于场景）：
   - 如果场景是【已有产品求改进】：对比竞品痛点，为我方产品提出至少3条“差异化竞争”与“迭代优化”的落地建议。
   - 如果场景是【无产品求入局】：基于当前评论反馈的满足度，评估该品类是否存在“未被满足的刚需”，并给出新产品冷启动的切入点建议。

【输出格式】
严格遵循 Markdown 格式输出，包含“核心痛点雷达图（数据表格形式）”、“产品亮点清单”与“基于当前场景的商业决策建议”。
```

### 4.2 泛分析师 Prompt 映射
```text
【角色设定】
你是一位顶级商业战略分析师和行业研究员，擅长从碎片化的互联网信息中拼凑出企业的全貌，并评估商业可行性。

【用户当前诉求场景】
{{ 场景变量：已有产品求改进 / 无产品求入局 }}

【输入数据】
基于搜索引擎抓取的竞品企业新闻、官网介绍、百科背景等广谱文本。

【执行任务（方法论映射）】
1. 发展里程碑（对应时间线回溯法）：梳理该竞品从创立至今的关键节点，形成时间线。
2. 基因与壁垒剖析（对应核心团队背景穿透）：分析其团队背景和核心资源，指出其“不可复制的竞争优势”是什么。
3. 动态战略输出（基于场景）：
   - 如果场景是【已有产品求改进】：梳理竞品当前的商业模式和主要获客渠道，寻找我方可以抢占的渠道空白。
   - 如果场景是【无产品求入局】：评估该领域的行业壁垒（技术/资金/渠道），并判断对于新入局者的难度评级。

【输出格式】
输出为结构化的商业分析简报，包含“发展历程”、“核心竞争力与壁垒模型”、“基于当前场景的入局/竞争策略”。
```

### 4.3 股市金融分析师 Prompt 映射
```text
【角色设定】
你是一位持有 CFA 证书的资深证券分析师，擅长透过财报数据看透企业真实的经营健康度与行业盈利空间。

【用户当前诉求场景】
{{ 场景变量：已有产品求改进 / 无产品求入局 }}

【输入数据】
竞品企业近三年的核心财务指标（营收、净利润、毛利率、研发费用率、经营性现金流）及最新股价动态。

【执行任务（方法论映射）】
1. 盈利质量评估（对应三表联动分析）：结合“经营性现金流”判断其利润支撑，指出潜在的坏账或现金流断裂风险。
2. 动态战略输出（基于场景）：
   - 如果场景是【已有产品求改进】：通过分析竞品的研发投入和库存周转率，判断其未来的发力方向和当前的销售疲软点。
   - 如果场景是【无产品求入局】：重点测算该品类头部企业的平均毛利率和资产周转率，评估新玩家入局的资金门槛和预期投资回报周期。

【输出格式】
输出专业财务诊断报告，必须包含“核心财务指标趋势分析”、“经营健康度诊断结论”、“基于当前场景的资金与风险评估”。
```

## 五、 数据源降级机制 (Fallback & Graceful Degradation)

由于系统强依赖于三路不同的数据源抓取（电商评论、全网搜索、金融API），在真实商业环境中，部分竞品可能因为“未上市”、“没有电商渠道”或“防爬虫策略”导致数据获取失败。因此，必须引入**柔性降级机制 (Graceful Degradation)**，确保分析任务不会因为单一节点失败而全盘崩溃。

### 5.1 维度一：Review 数据降级策略
- **正常情况**：成功抓取亚马逊、京东等电商平台的结构化 Review 和 Q&A 数据。
- **降级触发**：该竞品是 B2B 企业无电商渠道，或爬虫被封禁。
- **降级动作**：Agent 自动切换至“社交媒体情绪分析模式”。调用通用搜索工具（Bing/DuckDuckGo），以 `竞品名 + "骗子" OR "坑" OR "怎么样"` 为关键词，抓取知乎、贴吧、Reddit 等社交媒体的非结构化吐槽，通过大模型从中提取痛点。
- **前端感知**：报告区提示“未获取到电商 Review，当前采用社交媒体舆情分析结果替代”。

### 5.2 维度二：泛分析数据降级策略
- **正常情况**：成功抓取企查查、百科、官网等深度结构化资料。
- **降级触发**：竞品为极其低调的初创团队，全网无词条。
- **降级动作**：退回到基础的“搜索引擎摘要（Snippet）总结模式”，基于搜索结果前 20 条的摘要片段，进行模糊的商业模式推演。同时明确在报告中标注“数据置信度低”。

### 5.3 维度三：股市金融数据降级策略
- **正常情况**：成功调用金融 API 获取该上市公司的三表数据和研发占比。
- **降级触发**：该竞品为非上市公司（未公开财报）。
- **降级动作**：
  - 第一步：Agent 尝试搜索该公司的“融资历程（如天眼查数据）”，以融资金额和轮次作为资本认可度的替代指标。
  - 第二步：如果融资数据也缺失，该维度**直接熔断（Circuit Breaker）**。
- **前端感知**：财务板块显示“该竞品为非上市公司，无法获取财报数据”，并自动折叠该模块，不强行瞎编数据（防止大模型幻觉）。

## 六、 后续开发指导

1. **后端开发 (Go/Eino)**：
   - 数据库模型（Entity）需按照 `Enterprise -> Project -> AnalysisEngine` 进行重构，并在 Project/Task 层面新增 `Scenario (业务场景)` 字段。
   - Eino Agent 的编排（Graph）需支持分支执行，针对三大脉络分别调用不同的子 Agent，并将用户诉求（Scenario）动态注入到 Prompt 模板中。
   - **实现数据降级逻辑**：在调用 Tool 时捕获错误，触发 Fallback Tool；或者在 Prompt 中明确规定“如果没有搜到数据，必须返回特定状态码，绝对禁止编造”。
2. **前端开发 (Vue3)**：
   - UI 层级需配合调整：新建分析时，必须让用户选择“当前诉求”（如：已有产品求改进 / 无产品求入局）。
   - 报告展示页需要支持 Tab 切换（产品Review视图 / 宏观战略视图 / 财务健康视图），并高亮展示“基于当前场景的商业决策建议”。
   - **降级 UI 支持**：对于熔断的模块，展示优雅的缺省图（如：“未上市企业，无公开财报”），而不是报错弹窗。
3. **数据源接入**：
   - 优先打通简单的 API 数据源，后续逐步引入 RPA 爬虫获取复杂的 Review 数据。
