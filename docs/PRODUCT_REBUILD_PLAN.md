# 竞品分析系统产品重构计划

> 本文档用于记录当前这轮产品与技术改造计划，目标是把现有“能生成长报告的 Demo”收成“业务、产品、销售能直接使用的竞品分析工具”。

## 一、总目标

- 只产出一份业务可直接使用的精炼综述报告，而不是三篇并列长稿。
- 强化 `Review` 作为第一主视角，宏观与财务只做补充判断。
- 将聊天助手升级为全局常驻智能入口，支持关闭、重新唤起、文本上传、自动填参和触发分析。
- 建立可信证据链，报告中同时展示原文摘录、来源名、地址链接与时间信息。
- 提升搜索能力与时间感知，避免只依赖训练记忆或默认停留在 2024 年的信息。
- 用真实 PDF 导出替换当前打印方案。

## 二、核心产品原则

### 2.1 主产物只有一份

- 系统最终默认只展示一份综述报告。
- 三个 Agent 的原始稿件不作为主阅读入口。
- 如果保留三维原稿，只作为折叠调试信息存在。

### 2.2 Review 永远是主视角

- 主报告先回答“这个产品哪里最痛、哪里最强、我们怎么打”。
- 宏观与财务内容只能补充，不得抢占主篇幅。

### 2.3 报告必须能打仗

- 报告必须服务于产品迭代、业务判断和销售话术。
- 禁止学术化、咨询化、竞品自夸视角。
- 结论必须落到“这周能做什么”。

### 2.4 聊天助手不是附属插件

- 聊天助手应为全局常驻入口。
- 用户可直接通过聊天描述需求，由助手理解、追问、填参并触发分析。

## 三、P0：报告输出重构

### 3.1 单综述报告结构

主报告统一改造成以下结构：

1. 一句话结论
2. 用户最痛 3 点
3. 竞品最强 3 点
4. 我们可攻击 3 点
5. 这周可执行动作
6. 补充背景

### 3.2 输出长度约束

- 主报告默认控制在 800 到 1500 字。
- 聊天助手默认回答控制在 150 到 300 字，除非用户明确要求展开。
- 禁止“先铺背景、后讲重点”的长篇结构。

### 3.3 用户语言化

- 技术结论必须翻译成用户一句话抱怨。
- 每个痛点都要能转成文案、卖点或攻击点。

## 四、P0：多 Agent 共建共享后再汇总

### 4.1 共享证据池

- Review、宏观、财务三 Agent 不再各写各的长稿。
- 三方先把搜索证据、中间结论、风险与机会写入共享证据池。

### 4.2 中间结论格式

每个 Agent 输出统一中间结构：

- 发现
- 风险
- 机会
- 证据

### 4.3 Master 只做最终汇总

- Master 读取共享证据池与中间结论。
- 最终只生成一份综述报告。
- 汇总时 Review 权重最高。

## 五、P0：搜索能力强化

### 5.1 查询改写

- 每个维度自动生成更具体 query。
- query 必须包含：产品词、场景词、年份词、来源词。
- 默认加入 `2025 2026 最新` 等时间约束。

### 5.2 来源分层

将搜索来源分层抓取：

- 官网与技术规格
- 媒体评测
- 电商评论
- 社区讨论
- 财报与公告

### 5.3 检索兜底

- 第一次检索失败时自动改写 query 再搜。
- 禁止检索失败后直接让模型凭记忆硬写。

### 5.4 去噪

- 过滤词典页、垃圾聚合页、无关百科页。
- 提高评测站、评论区、官方站、财报站优先级。

## 六、P0：时间感知

### 6.1 当前时间注入

- 在系统 Prompt 中显式写入当前日期、当前年份、当前季度。
- 检索 query 默认附带当前年与近两年窗口。

### 6.2 实时信息约束

- 最新信息必须来自搜索结果。
- 模型不得用训练记忆冒充实时动态。
- 使用历史信息时必须标注为历史数据。

## 七、P0：证据链展示

### 7.1 报告中直接展示证据

每个关键结论下都应展示：

- 原文摘录
- 来源名
- 跳转地址
- 发布时间（如果可获取）

### 7.2 证据卡片化

证据以卡片形式展示，结构为：

- 结论
- 原文摘录
- 来源
- 链接

这样报告不仅“有结论”，还能“自证结论”。

## 八、P0：聊天助手升级

### 8.1 全局常驻

- 聊天助手不再只存在于报告页。
- 登录页、工作台、分析页、报告页都能唤起。

### 8.2 可关闭与自适应扩展

- 右侧助手必须可关闭。
- 助手关闭后，中间内容区自动扩展并吃满剩余空间。
- 助手重新打开时恢复对应布局。

### 8.3 右侧只服务聊天

- 右侧一旦展开，只保留聊天助手主界面。
- 不再混放执行过程、无意义说明或冗余信息。

### 8.4 简明模式

聊天助手默认回答结构：

- 先结论
- 再 3 条要点
- 最后动作建议

## 九、P0：聊天驱动分析

### 9.1 自然语言理解

用户可以直接输入：

- 要分析哪个竞品
- 想看什么重点
- 属于哪个项目
- 是已有产品优化还是准备入局

### 9.2 缺参追问

- 参数不全时，助手主动追问。
- 参数齐全时，自动生成分析任务。

### 9.3 自动填参与触发

- 助手识别出的竞品、项目、场景、目标自动写入任务结构。
- 用户确认后由助手直接触发分析。

## 十、P0：文本上传能力

### 10.1 首批支持格式

- txt
- md
- pdf
- docx
- csv
- json

### 10.2 上传后的用途

- 作为问答上下文
- 辅助提取竞品与分析参数
- 辅助触发分析
- 作为证据补充来源

## 十一、P0：竞品图片与截图能力

### 11.1 图片抓取

- 抓取竞品官网主图
- 抓取商品主图
- 抓取详情页卖点图
- 抓取宣传图与关键截图

### 11.2 图文联动

- 图片不能单独堆在一起。
- 要嵌入综述报告中的关键结论位置。
- 每张图需要附带图注说明：
  - 这张图在卖什么
  - 用户会被什么打动
  - 我们可以从哪里攻击

## 十二、P0：真实 PDF 导出

### 12.1 替换打印方案

- 废弃 `window.print()` 假导出。
- 使用真正的 PDF 生成方案。

### 12.2 导出要求

- 只导出综述报告主内容与关键证据。
- 不导出聊天区与执行日志。
- 支持 A4 分页、页边距、封面和页头信息。

## 十三、P0：前端布局改造

### 13.1 三栏可调

- 左侧菜单、中间内容、右侧助手支持拖拽缩放。
- 栏宽状态应支持持久化。

### 13.2 助手优先空间

- 右侧助手默认不应过窄。
- 助手应拥有足够的聊天和上传空间。

### 13.3 滚动隔离

- 主报告
- 聊天消息
- 上传材料
- 执行过程

以上区域需要独立滚动，避免相互撑开。

## 十四、P1：质量观测与回归

### 14.1 基准样例集

建立固定竞品集回归测试，例如：

- Mac mini
- DJI Osmo Action
- GoPro
- Insta360

### 14.2 验收维度

每次改动后检查：

- 是否命中 2025/2026 信息
- 是否展示原文摘录和地址
- 是否只产出一份综述报告
- 是否突出 Review 主视角
- 是否提供可执行动作建议
- 聊天回答是否简洁可用

### 14.3 失败分类

对失败情况做标准分类：

- 搜不到
- 搜偏了
- 时间旧
- 来源垃圾
- 模型乱补
- 输出太长

## 十五、建议实施顺序

### 阶段 1：先把结果收短

- 改 Master 汇总 Prompt
- 改三 Agent 中间输出结构
- 落地单综述报告

### 阶段 2：把搜索和时间感知打牢

- 改 query 生成
- 注入当前日期
- 做检索兜底
- 展示证据摘录与链接

### 阶段 3：升级聊天助手

- 做全局常驻入口
- 做关闭与自动扩展
- 做简明模式
- 做自然语言填参与触发分析

### 阶段 4：加强材料与视觉信息

- 做文本上传
- 做竞品图片抓取
- 做截图与图注

### 阶段 5：补齐输出链路

- 做真实 PDF 导出
- 做回归样例与质量观测

## 十六、验收标准

- 默认只看到一份综述报告，不再三篇长稿并列。
- Review 是主角，宏观与财务只补充。
- 报告中能直接看到原文摘录、来源名、链接。
- 聊天助手可关闭、可重新打开、可上传文本材料。
- 用户通过聊天即可触发分析，不必先手填表单。
- PDF 导出是真导出，不是打印。
- 右侧助手关闭后，中间内容区自动扩展。

---

如需推进实现，建议从“单综述报告 + 搜索证据链 + 聊天助手常驻入口”三项开始落地。
